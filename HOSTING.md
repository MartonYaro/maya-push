# MAYA Push — Hosting & Production Checklist

## TL;DR — Free → Paid migration path

| Step | What | Where | Cost |
|------|------|-------|------|
| 1 | Render Starter plan ($7/mo) — no cold starts, custom domains, persistent disk | Render dashboard → Service → Plan | $7/mo |
| 2 | 10 GB persistent disk for SQLite | uncomment `disk:` in `render.yaml` + redeploy | $1/mo |
| 3 | Custom domain | Render → Settings → Custom Domains | free (own domain) |
| 4 | Resend with own domain (no rate limit) | resend.com → Domains | free up to 3k/mo |
| 5 | Telegram bot for ops alerts | @BotFather | free |
| 6 | Postgres (when SQLite outgrows ~1M rows) | Render → New Postgres | $7/mo |

Total to start production: **~$8–15/mo**.

---

## 1. Upgrading Render to Starter

1. Render dashboard → `maya-push` service → **Settings → Plan → Upgrade to Starter**.
2. Settings → **Custom Domains** → add `app.yourdomain.com` → set CNAME at registrar.
3. Render auto-issues TLS via Let's Encrypt (~2 min).

After upgrade you also get:
- No cold starts (free plan sleeps after 15 min idle)
- 512 MB → 2 GB RAM headroom
- Persistent disk support (next step)

## 2. Persistent disk for SQLite

The free plan uses `/tmp` which is wiped on every deploy. Steps:

1. Edit `render.yaml`: uncomment the `disk:` block.
2. Change `DB_PATH` env var: `/tmp/maya.db` → `/var/data/maya.db`.
3. Commit & push. Render attaches a 10 GB SSD and reuses it across deploys.

⚠ **First migration**: existing `/tmp/maya.db` content is lost. If you have prod data, dump first:
```sh
# in Render Shell
sqlite3 /tmp/maya.db ".backup /tmp/dump.db"
# download via Render Shell → upload to /var/data after disk attached
```

## 3. Required environment variables

Set in **Render → Environment**:

| Var | Required | Purpose |
|-----|----------|---------|
| `JWT_SECRET` | yes | Auto-generated by Render Blueprint |
| `APPTWEAK_API_KEY` | yes | AppTweak API |
| `PUBLIC_URL` | yes | `https://app.yourdomain.com` — used in email links |
| `DB_PATH` | yes | `/var/data/maya.db` (paid) or `/tmp/maya.db` (free) |
| `NODE_ENV` | yes | `production` (enables HSTS) |
| `RESEND_API_KEY` | recommended | Without it emails go to console log |
| `EMAIL_FROM` | recommended | `MAYA Push <noreply@yourdomain.com>` once Resend domain verified |
| `TELEGRAM_BOT_TOKEN` | optional | Admin notifications |
| `TELEGRAM_ADMIN_CHAT` | optional | Your chat_id from @BotFather flow |
| `ALLOW_ORIGIN` | optional | Set to `https://app.yourdomain.com` to lock CORS |

## 4. Resend (transactional email)

1. Sign up at https://resend.com (free 100/day, 3k/mo).
2. **Domains** → add `yourdomain.com` → add the SPF/DKIM/DMARC records at your DNS.
3. Copy API key → set `RESEND_API_KEY` in Render.
4. Update `EMAIL_FROM` to `MAYA Push <noreply@yourdomain.com>`.

Without a verified domain you can only send from `onboarding@resend.dev` and only to your own verified address.

## 5. Telegram ops bot

1. Telegram → @BotFather → `/newbot` → save token.
2. Send any message to your bot.
3. Open `https://api.telegram.org/bot<TOKEN>/getUpdates` → copy `chat.id`.
4. Set `TELEGRAM_BOT_TOKEN` and `TELEGRAM_ADMIN_CHAT` in Render.

You'll get notifications for: new signups, new top-up requests.

## 6. When to migrate from SQLite to Postgres

SQLite on a persistent disk is fine until any of:

- ≥10 concurrent writers (we currently have 1: the cron tick)
- DB size > ~5 GB
- You need read replicas or point-in-time recovery
- You go multi-region

When you hit that, the migration is mechanical:
- Replace `better-sqlite3` with `pg` + write a thin wrapper exposing the same `db.prepare(...).run/get/all` shape (or switch to Drizzle/Knex).
- All existing SQL is ANSI-compatible except: `INSERT OR IGNORE` → `ON CONFLICT DO NOTHING`, `INTEGER PRIMARY KEY AUTOINCREMENT` → `BIGSERIAL`.

- ✅ Audit log (logins, signups, password changes)

```sh
# 4. Check audit log in admin panel: /admin
```

## 9. Backups (paid plan)

Render Starter+ disks support snapshots, but not automated. Recommended cron job:

```sh
# add to server/scripts/backup.js — runs daily, uploads .db to S3/R2
sqlite3 /var/data/maya.db ".backup /var/data/backup-$(date +%F).db"
```

For full disaster recovery, switch to Render Postgres — automated daily backups, 7-day PITR.
